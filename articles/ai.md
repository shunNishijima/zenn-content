# AIエンジニアが2種類に分かれる時代、あなたのキャリアはどちらを目指すべきか

**この記事の要点**

- AI人材は「AIを作る人（Builder）」と「AIで作る人（User）」の2層に収束しつつある
- 日本の多くのエンジニアが向かうべきは後者だが、後者の中にも「高付加価値層」と「代替可能層」の差が生まれている
- 2025〜2026年時点でキャリアを守り、伸ばすためには「AIとドメイン知識の掛け算」が最重要スキルになる

---

2026年現在、転職サービスや技術コミュニティで「AI人材」という言葉が溢れています。しかしその実態を観察すると、同じ「AI人材」でも全く異なるスキルセットと市場価値を持つ2つの層に明確に分かれてきています。この記事では、AIエンジニアの2種類のタイプを整理し、日本のエンジニアとして今どちらを目指すべきかを、具体的なスキルマップとキャリアパスとともに解説します。

---

## AIエンジニアを2種類に分ける「シンプルな軸」

まず最初に結論を述べると、AI人材は次の2種類に大別できます。

| タイプ | 通称 | 主な役割 |
|--------|------|----------|
| **Foundation Builder** | 基盤を作る人 | LLM・拡散モデルの学習・アーキテクチャ設計・推論最適化 |
| **Application Builder** | 価値を作る人 | AIを使って製品・業務・プロセスを変革するエンジニア |

前者は GoogleやAnthropic、OpenAIのような基盤モデルを研究・開発する側の人材です。後者は、GPT-4oやClaude Sonnet、Geminiといった既存の強力なモデルをAPIやツールとして活用し、実際のプロダクトや業務に価値を埋め込む人材です。

「どちらが上か」ではなく、「**自分がどちらに適しているか**」「**市場がどちらを求めているか**」を正確に把握することが、今のキャリア判断で最も大切です。

---

## Foundation Builder：基盤を作る人の現実

### 求められるスキル

- 深層学習理論（Transformer・拡散モデルの仕組みを数式レベルで理解できる）
- 大規模分散学習の設計（FSDP、Megatron-LMなどを扱える）
- CUDA・Tritonなどを使ったカーネル最適化
- 論文実装力（arXivの最新論文を読んで再現できる）

### 日本でのポジション数と現実

正直に言うと、Foundation Builderのポジションは**日本国内ではほぼ存在しない**のが現状です。サイバーエージェントやNTT、富士通といった国内大手が独自のLLM開発を進めてはいますが（例：CyberAgent の CyberAgentLM、NTT の tsuzumi）、採用数は数十人〜数百人規模。競争倍率は非常に高く、かつ博士号保有者や国際論文の実績が実質的な要件になります。

いわば「プロ野球選手」に近い世界です。明確な素質・実績がある人は全力で目指す価値がありますが、そうでない場合は別の戦略を取るべきでしょう。

## Application Builder：価値を作る人の現実と「罠」

### なぜここが注目されるか

ChatGPTやClaude、Geminiが汎用的に使えるようになった今、**AIを"呼び出す"こと自体はコモディティ化**しています。Pythonで`anthropic.Anthropic().messages.create()`を叩くだけなら、誰でも数時間で習得できます。

```python
import anthropic

client = anthropic.Anthropic()
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "こんにちは"}]
)
print(response.content[0].text)
```

問題は、「APIを叩ける」だけの人が溢れた結果、Application Builderの中にも**高付加価値層と代替可能層**の格差が生まれていることです。

### 代替可能層に落ちるエンジニアの特徴

- ChatGPTのラッパーを作るだけで「AIアプリ開発者」を名乗っている
- プロンプトを工夫することが主な仕事になっている
- ドメイン知識（業界・業務・ユーザーの文脈）をほぼ持っていない
- RAGやエージェントのコピー実装はできるが、なぜそれで解けるかを説明できない

実際に私が周囲のエンジニアを観察した感覚でも、2024年後半から「AIエンジニアとして転職できた」という事例が急増した一方で、1年以内に「期待されていたほど価値を出せなかった」と感じているケースも少なくないようです。

### 高付加価値層に入るエンジニアの特徴

| スキル領域 | 具体的な能力 |
|----------|------------|
| **AIの仕組み理解** | RAG・Function Calling・Agent Loopのトレードオフを説明できる |
| **評価設計** | LLMの出力品質を定量的に測るevalを自分で設計できる |
| **ドメイン知識** | 医療・法務・金融など特定領域の業務フローをAIで再設計できる |
| **システム設計** | AIをプロダクションで動かすための信頼性・コスト・レイテンシの設計ができる |
| **プロダクト思考** | 「AIでできること」と「ユーザーが本当に困っていること」を両方理解している |

## 今のトレンドが示す「掛け算」の重要性

2025〜2026年にかけて特に顕著なのは、**AIスキル × ドメイン知識**の組み合わせを持つ人材への需要急増です。

例えば：

- **医療 × AI**：電子カルテのサマリー自動生成、診断支援ツール開発
- **法務 × AI**：契約書レビュー、規制対応のドキュメント検索
- **製造 × AI**：設備点検の異常検知、工程最適化のシミュレーション

これらの領域では、純粋なAIエンジニアだけでも、純粋な業界専門家だけでも解決できません。両方を橋渡しできる人材が、現時点で最も市場価値が高い層になっています。

転職エージェントの情報（求人票ベースで観察）によると、「業務AIエンジニア」「AIプロダクトエンジニア」といった職種の求人が2024年比で大幅に増加しているようです（具体的な数値はTBDですが、体感では明らかな増加傾向です）。

## 具体的なスキルアップロードマップ

### Phaseごとの推奨行動

**Phase 1（今すぐ〜3ヶ月）：AIの基礎を"使える"レベルに**

1. Anthropic APIまたはOpenAI APIでシンプルなRAGを実装する
2. LangChainやLlamaIndex等のフレームワークを1つ素のAPIと比較しながら学ぶ
3. MCP（Model Context Protocol）の概念を理解し、簡単なMCPサーバーを立てる

**Phase 2（3〜6ヶ月）：評価と本番運用を学ぶ**

1. LLMの出力をどう評価するかを設計する（精度・コスト・レイテンシ）
2. Promptfoo等の評価ツールを使いこなす
3. 本番でのAI導入事例（失敗含む）を読み込む

**Phase 3（6ヶ月〜）：ドメインとの掛け算を意識する**

1. 自分のこれまでの業務経験（業界・職種）でAIが解けそうな課題を定義する
2. その課題に対してプロトタイプを作り、LT・ブログ・OSSで外部発信する
3. 特定ドメインに特化したポジション（業界特化のスタートアップや部門）を意識して転職・社内異動を検討する

## まとめ：「どちらか」よりも「自分がどこで戦えるか」

Foundation BuilderとApplication Builderの二択は、優劣の問題ではありません。ただ、日本のエンジニア市場の現実を見ると、大多数の人にとっての現実的なキャリアパスはApplication Builderです。

その中で生き残るには、「AIを使える」だけでは不十分で、**AIとドメイン知識を組み合わせて業務・製品の問題を解ける人**になることが求められています。

まず自分の強みとなる業界・業務領域を1つ明確にする。その上でAIの実装スキルを積み上げる。この順序で動くことが、今の時代最も合理的なキャリア戦略だと考えています。

## よくある質問

**Q. 数学が得意ではないとAIエンジニアになれませんか？**

Application Builderとしてのキャリアであれば、深い数学の知識がなくても十分に活躍できます。Transformerの数式を理解している必要はなく、「どんな入力をすると何が起きるか」「なぜ失敗するか」を経験的に理解できるエンジニアリング力の方が実務では重視されます。Foundation Builderを目指す場合は線形代数・確率統計・最適化の知識が必須になります。

**Q. 今からLLM・生成AIを学ぶのは遅いですか？**

遅くはありません。ただし「APIを叩けるようになる」だけでは差別化になりません。2026年時点でのスタートなら、基礎的なAPI活用学習に時間をかけすぎず、自分のドメイン知識と組み合わせてどんな問題を解けるかをすぐに試した方が、キャリア上の成果が出やすいです。

**Q. AIエンジニアとMLエンジニアの違いは何ですか？**

MLエンジニア（Machine Learning Engineer）は伝統的に「特徴量エンジニアリング・モデルのトレーニング・パイプライン構築」を指していました。AIエンジニアはより広義で、LLMのAPIを活用したアプリケーション開発も含みます。最近の求人では両者の境界が曖昧になっており、職種名よりもJDの具体的な業務内容を確認することをおすすめします。
